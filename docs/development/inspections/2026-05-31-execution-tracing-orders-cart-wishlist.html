Reporte de tracing: órdenes, carrito y wishlist

HealthBytes · QA y Observabilidad · 2026-05-31 · Inspección estática del código actual

Alcance: hechos observables en código. El estado Zustand está en frontend/store/; no existe una carpeta llamada Frozen. No se incluyen propuestas, refactors ni inferencias de rendimiento.

0. Infraestructura común de recepción

  1. FastAPI registra routers con prefijos reales: /orders, /cart y /favorites. backend/app/main.py:363-370
  2. Las rutas dependen de get_current_user. Antes del router, middleware attach_user_for_rate_limiting intenta extraer Authorization: Bearer ..., verificar Clerk o JWT legado y cargar User. Si encuentra usuario, lo asigna a request.state.user. backend/app/main.py:107-165
  3. get_current_user retorna primero request.state.user si existe. Si no existe, sigue sus ramas de autenticación: bypass dev configurado, Clerk o JWT legado. backend/app/middleware/auth.py:126-226
  4. get_db entrega una AsyncSession SQLAlchemy y la cierra al terminar la petición. PostgreSQL usa URL transformada a postgresql+psycopg://. backend/app/db/database.py:9-44

1. Carga y procesamiento de vista de órdenes

Dato exacto: esta vista no consulta tabla payments ni construye una categoría frontend llamada paid. Expone filtros sobre orders.status: all, unpaid, processing, shipped, delivered y returns. frontend/app/orders.tsx:138-145

1. Interacción UI

  1. Al montar OrdersScreen, useEffect espera isSignedIn e isLoaded. Calcula status desde selectedFilter, ejecuta getToken() y, si hay token, llama fetchOrders(token, 0, undefined, status). frontend/app/orders.tsx:147-211
  2. Al presionar filtro, handleFilterPress(filterId) actualiza selectedFilter, obtiene token y llama fetchOrders(token, 0, undefined, status). Para all, status queda undefined. frontend/app/orders.tsx:226-235
  3. Pull-to-refresh ejecuta onRefresh. Botón “Cargar más órdenes” obtiene token y llama loadMoreOrders(token). frontend/app/orders.tsx:213-222,275-305

2. Mutación Zustand

  1. useOrders.fetchOrders actualiza estado antes de red: { isLoading: true, error: null, currentStatus: status ?? null }. frontend/store/orderStore.ts:43-51
  2. En éxito reemplaza lista: orders, hasMore: orders.length === 50, isLoading: false. En error conserva lista previa y escribe error e isLoading: false. frontend/store/orderStore.ts:52-63
  3. loadMoreOrders evita nueva ejecución si isLoadingMore ya es verdadero. Antes de red escribe { isLoadingMore: true, error: null }; en éxito concatena resultados; en error escribe mensaje e isLoadingMore: false. frontend/store/orderStore.ts:69-90

4. Recepción FastAPI y lógica

  1. GET /orders llega a list_orders(skip, limit, status_filter, db, current_user). Router valida skip >= 0, 1 <= limit <= 50 y mapea query status a status_filter. backend/app/api/v1/orders.py:120-127
  2. Router llama order_service.list_orders_for_request. Service inicia select(Order); admin no agrega filtro por usuario; seller y resto agregan Order.user_id == current_user.id. Si llega status_filter, agrega Order.status == status_filter. backend/app/services/order_service.py:172-189
  3. Service agrega selectinload(Order.items), orden descendente por created_at e id, offset(skip) y limit(limit); ejecuta statement y retorna órdenes. backend/app/services/order_service.py:190-198
  4. Router transforma cada orden y cada item a OrderResponse y OrderItemResponse. En excepción no HTTP, responde HTTP 500 con ErrorResponse.server_error. backend/app/api/v1/orders.py:145-182

6. Retorno y render

  1. FastAPI retorna HTTP 200 con arreglo JSON de órdenes. Cada objeto contiene id, user_id, created_at, status, address_id, payment_method e items. backend/app/schemas/order.py:65-93
  2. Zustand reemplaza o concatena orders. OrdersScreen pasa lista a FlatList. OrderListItem normaliza status, suma cantidades y calcula total desde item.price * item.quantity. frontend/app/orders.tsx:224-251,343-359; frontend/components/OrderListItem.tsx:19-45

  2. Botón se deshabilita si isRemoving contiene ID del producto. frontend/components/CartItem.tsx:24-30,139-150

2. Mutación Zustand

  1. removeProduct(productId, getToken) lee removingProducts. Si mismo ID ya existe en Set, retorna sin request. frontend/store/cartStore.ts:318-324

  3. El Set es por ID. Invocaciones para IDs distintos atraviesan función independientemente y construyen requests separados. frontend/store/cartStore.ts:318-350

3. Presiones repetidas de Like / Wishlist

1. Interacción UI

  1. FavoriteButton aparece en tarjeta de producto y detalle. frontend/components/ProductCard.tsx:267; frontend/app/product/[id].tsx:729-732
  2. Pressable.onPress evita propagación y llama handlePress(). handlePress obtiene token; sin token muestra Alert y retorna. Con token ejecuta animación si corresponde y espera toggleFavorite(productId, getToken). frontend/components/FavoriteButton.tsx:30-48,62-93
  3. La fila de wishlist tiene otra entrada: WishlistTableRow.handleRemove obtiene token y llama mismo toggleFavorite. frontend/components/WishlistTableRow.tsx:25-29,165-170

  3. En excepción, store hace rollback mediante set({ favoriteIds }), usando referencia capturada al inicio de esa invocación. frontend/store/favoritesStore.ts:49-52

6. Retorno y procesamiento Zustand

  1. POST exitoso retorna HTTP 201 con FavoriteResponse, incluyendo producto opcional. Store no usa payload; conserva Set optimista. backend/app/api/v1/favorites.py:19; backend/app/schemas/favorite.py:17-31; frontend/store/favoritesStore.ts:42-48
